# Rule Plan IR v1

## Purpose

`game.rule_plan_v1` is a canonical, versioned optimization contract emitted by both the JavaScript compiler/exporter and the native C++ compiler.

It is not the full executable rule representation. It is the shared metadata layer used to:

- Compare JS and C++ lowering before gameplay replay.
- Move safe scan/replacement optimizations into compiler output.
- Keep optimization work parity-friendly by adding metadata first, then teaching each runtime to consume it.

The live JavaScript compiler/runtime also attaches a small behavior-neutral `rulePlanMetadata` object to each compact `Rule` instance. That live metadata mirrors the conservative shape/classification fields from `rule_plan_v1` and is intentionally not used for execution until a measured optimization earns its way in.

The current parity gate is:

```sh
make rule_plan_parity_tests
```

That target compares `rule_plan_v1` for every unique source in `src/tests/resources/testdata.js` where both compilers can emit IR. The simulation corpus currently has `469` replay test cases over `453` unique source texts; repeated sources are compared once here because `rule_plan_v1` is compile output, not per-input replay output. The simulation corpus contains playable legacy/janky games that may carry JS diagnostics, so `document.error_count` is not treated as a skip condition for this structural parity gate.

Compiler diagnostics remain covered by:

```sh
make compilation_tests_cpp
```

## Top-Level Shape

```json
{
  "schema_version": 1,
  "rules": [[{ "...": "rule plan" }]],
  "late_rules": [[{ "...": "late rule plan" }]]
}
```

`rules` and `late_rules` preserve executable group boundaries after impossible-rule filtering. Rule entries inside a group are canonicalized by the parity harness before comparison, so the gate is insensitive to harmless intra-group emission-order differences while still checking structure.

## Rule Entry

Each rule entry contains stable source/runtime identity plus precomputed scan metadata:

- `rule_index`: Index inside the emitted canonical group.
- `group_index`: Index inside the emitted canonical rule-group list.
- `late`: Whether this came from the late-rule phase.
- `direction`: Direction bitmask using existing PuzzleScript constants.
- `line_number`: Source line used for diagnostics/debugging.
- `group_number`: Original rule group number from lowering.
- `rigid`: Whether rigid rollback semantics apply.
- `is_random`: Whether random rule-group selection semantics apply.
- `has_replacements`: Whether the executable rule has any RHS replacement work.
- `has_ellipsis`: Whether any row contains an ellipsis.
- `row_count`: Number of pattern rows.
- `has_commands`: Whether the rule queues output commands.
- `command_names`: Command names in lowered order, without arguments.
- `simple_deterministic_row_rule`: Conservative fast-path classification for one-row, non-random, non-rigid, non-ellipsis rules with no output commands.
- `delta_hint`: `{ "dx": number, "dy": number }` for scan direction.
- `rule_object_ids`: Sorted object ids required anywhere by the rule LHS.
- `rule_movement_bits`: Sorted `[layer, movement_bit]` pairs required anywhere by the rule LHS.
- `rows`: Per-pattern-row metadata.
- `replacements`: Per-cell replacement classification metadata.

`rule_object_ids` and `rule_movement_bits` are safe for whole-rule prefilters: if the current board lacks those objects or movement bits, the rule cannot match.

## Row Entry

Each `rows[]` entry describes one row of a rule pattern:

- `row_index`: Row index in the rule pattern.
- `ellipsis_count`: Number of ellipses in this row.
- `object_ids`: Sorted object ids required by this row.
- `movement_bits`: Sorted `[layer, movement_bit]` pairs required by this row.
- `concrete_anchor_object_ids`: Sorted concrete object ids that may anchor candidate scanning.
- `any_anchor_object_ids`: Array of sorted object-id choices for property/any-object anchors.
- `concrete_cell_count`: Number of non-ellipsis cells in the row.
- `min_concrete_suffix`: Number of concrete cells after the final ellipsis.
- `scan_order`: Canonical scan order string. Current value is `x_major`.

Row metadata is intended for line-level rejection, anchor selection, and ellipsis planning. It must not change PuzzleScript scan semantics; it only describes safe ways to avoid work before falling back to the canonical matcher.

## Replacement Entry

Each `replacements[]` entry describes a non-guaranteed-noop replacement cell:

- `row_index`: Pattern row containing the replacement.
- `cell_index`: Pattern cell containing the replacement.
- `touches_objects`: Replacement may change object bits or choose a random object.
- `touches_movements`: Replacement may change movement bits or choose a random direction.
- `touches_movements_layer`: Replacement clears movement state for one or more layers.
- `touches_random`: Replacement consumes RNG through random object or random direction.
- `touches_random_entity`: Replacement contains random object/entity selection.
- `touches_random_dir`: Replacement contains random direction selection.
- `touches_rigid`: Replacement may affect rigid movement bookkeeping.
- `simple_direct_mask`: Conservative fast-path classification for non-random, non-rigid direct mask replacement.
- `objects_clear_ids`: Sorted object ids cleared directly.
- `objects_set_ids`: Sorted object ids set directly.
- `movements_clear_bits`: Sorted `[layer, movement_bit]` pairs cleared directly.
- `movements_set_bits`: Sorted `[layer, movement_bit]` pairs set directly.
- `movements_layer_bits`: Sorted `[layer, movement_bit]` pairs cleared by layer reset.
- `random_dir_bits`: Sorted `[layer, movement_bit]` pairs available to randomdir.
- `random_entity_object_ids`: Sorted object ids available to random entity replacement.
- `random_entity_choices`: Deterministic object choices for random entity replacement.
- `random_dir_layers`: Deterministic layer choices for random direction replacement.
- `movement_layers`: Sorted layers whose movement state is reset.

Replacement metadata is intended for specialized fast paths. A runtime may use it to choose a simple direct-mask kernel only when the flags prove that random, rigid, audio, and complex movement behavior are absent.

## Parity Rules

- Add metadata to both emitters before consuming it in either runtime.
- Keep fields canonical: sorted ids, sorted movement pairs, deterministic group/row ordering, and no platform-width-dependent mask words.
- Do not include raw 32-bit or 64-bit mask words in `rule_plan_v1`; ids and `[layer, movement_bit]` pairs are easier to compare across JS and C++ mask widths.
- A metadata field may be advisory for performance, but if present it must be semantically correct.
- If a runtime optimization based on rule-plan metadata fails, the fallback must be the existing canonical rule matcher.
