# Shared JS/C++ Compiler Optimization Plan

## Summary
- Keep C++/JS parity as the hard gate by adding a versioned, canonical **Rule Plan IR** generated by both compilers and compared in tests.
- Move safe runtime optimizations into compile/lower output so both JS and C++ can use identical metadata: anchors, row masks, movement masks, ellipsis plans, replacement classes, and scan bounds.
- Priority is work reduction, not clever micro-tuning: fewer candidate cells, fewer pattern tests, fewer full-row scans, fewer replacement branches.

## Key Changes
- Add a `rule_plan_v1` section to both JS and C++ compiled output with canonical serialization and hash comparison.
- Include per-rule/per-row metadata: direction delta, scan bounds, row object mask, row movement mask, concrete object anchors, property/any-object anchors, ellipsis concrete-cell counts, min suffix counts, and deterministic scan order.
- Include per-replacement metadata: simple/non-simple classification, touches objects, touches movement, touches random, touches rigid, touches audio, layer clear masks, random choice lists, and direct clear/set masks.
- Add movement-aware rule skipping in both engines using a precomputed `ruleMovementMask`, not just `ruleMask`.
- Add a shared “simple deterministic row rule” execution path: no random, no rigid, no ellipsis, one row, no debug logging. JS can generate this as specialized code; C++ can route to a flat kernel.
- Port the C++ anchor wins back to JS: concrete anchors, property/any-object anchors, anchor density cutoff, and deterministic sort/unique where needed.
- Port the C++ ellipsis line-prefilter to JS and then make both compilers emit ellipsis plan metadata rather than recomputing it at runtime.

## Parity Safeguards
- Add a test mode that compares JS and C++ `rule_plan_v1` before gameplay simulation; fail on the first structural mismatch.
- Keep current simulation and diagnostics gates mandatory: `simulation_tests_cpp`, `compilation_tests_cpp`, and JS parity tests remain non-negotiable.
- Add a debug differential mode that reports first divergence at rule-group/row-plan level: rule id, group id, row index, candidate starts, chosen random candidate, board string, movement masks, and RNG state.
- Every optimization lands behind plan metadata first, then runtime usage second, so semantic drift is caught before behavior changes.

## Suggested Implementation Order
1. Define and emit `rule_plan_v1` from JS `rulesToMask` and C++ lowering, including canonical JSON output for tests.
2. Add rule-plan parity tests over `testdata.js` and `errormessage_testdata.js` where compilation succeeds.
3. Move C++’s existing anchor metadata into `rule_plan_v1`, then teach JS to consume the same anchor plan.
4. Add shared ellipsis plan metadata and port the impossible-line skip to JS.
5. Add movement mask rule skipping in both engines.
6. Add simple-rule replacement classification and specialized simple-rule kernels in both engines.
7. Only after that, consider stride-specialized bit operations: JS generated code for `STRIDE_OBJ/MOV == 1/2`, C++ templated kernels selected from rule-plan width.

## Test And Metrics
- Correctness gates: C++ simulation `469/469`, C++ diagnostics `273/273`, JS parity replay `469/469`.
- New parity gate: `rule_plan_v1` canonical hash match for every shared successful compile case.
- Metrics to track for both JS and C++: total sim time, replay time, rules visited, candidates tested, pattern tests, row scans, ellipsis scans, replacements attempted/applied.
- Success target: C++ single-thread replay under `1s`; JS sim-only meaningfully improved by the same compiler metadata rather than one-off runtime hacks.

## Assumptions
- We do not change PuzzleScript semantics or diagnostics to gain speed.
- Compile-time normalization that changes warnings/errors, such as aggressive `no` simplification, is deferred until rule-plan parity is stable.
- JS remains 32-bit bitvec based; C++ remains 64-bit by default with 32-bit parity mode preserved.
